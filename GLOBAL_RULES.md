# AI開発グローバルルール

> このファイルはすべてのプロジェクトに共通するルールを定義する。  
> プロジェクト固有のルールは `RULES.md` に記載し、本ファイルを継承する。

---

## 0. このルールの使い方

### 基本方針

**すべてのルールの適用を推奨する。** ただしプロジェクトの規模・性質によって現実的でない項目がある場合は、`PROJECT_CONFIG.md` に採用・除外を明示したうえで判断してよい。

除外する場合も「適用しない」と明記することで、意図的な省略であることをエージェントが認識できる。

### プロジェクト規模の定義

| 規模 | 目安 |
|---|---|
| 🟢 小規模 | 単発スクリプト・数ファイル・使い捨てツール |
| 🟡 中規模 | 機能が複数・セッションをまたぐ・ファイル数10〜30程度 |
| 🔴 大規模 | ファイル数が多い・複数モジュール・リリース済みまたは継続開発 |

### 各ルールの推奨規模タグ

各セクションに以下のタグを付与している。タグは **推奨** であり、`PROJECT_CONFIG.md` で上書きできる。

| タグ | 意味 |
|---|---|
| `[全規模]` | 規模に関わらず適用を推奨 |
| `[中規模以上]` | 小規模では省略してもよい |
| `[大規模推奨]` | 中規模以下では状況に応じて判断する |

### エージェントの起動手順

プロジェクト開始時に以下を実行する

```
1. \MainSkills\SKILL.md を読む（SkillManager起動）
2. \MainSkills\skills\skill_index.yaml を読む
3. generate_index.bat を実行して index.yaml を最新化する
   → 実行できない場合：既存の index.yaml をそのまま使う
4. index.yaml をコンテキストに読み込む
5. PROJECT_CONFIG.md を確認する（なければ作成する）
6. 採用ルールに従って開発を進める
```

`PROJECT_CONFIG.md` のテンプレートは `PROJECT_CONFIG_TEMPLATE.md` として同梱する。

## 1. 基本方針 `[全規模]`

- 本ルールは **人間開発者とAIエージェントの共通ルール** とする
- 人間とAIは **同一のルール・同一の構造・同一の探索方法** を使用する
- すべてのコードは以下の原則を満たすこと

```
Human Readable  ← 人間が理解できない構造を禁止する
AND
AI Searchable   ← AIが探索できない構造を禁止する
```

- すべてのコードは **自己説明型（Self-Describing Code）** を目指す
- AIが探索しやすいよう 検索用コメントと機能インデックスを必須とする

---

## 2. アーキテクチャ原則

### 2-1. モジュール設計 `[全規模]`

- アプリケーションはモジュール単位で構成する
- 各モジュールは単一の責務を持ち、独立して動作できること
- モジュールの先頭には必ず役割・権限・依存関係を明記する（→ セクション5）
- モジュール追加後は インデックスを再生成する（→ セクション6）
- コアは最小限に保ち、機能はモジュールに委譲する

### 2-2. モジュール間インターフェース `[中規模以上]`

- モジュール間でデータをやり取りする場合は **JSON互換形式** を使用する
- プロジェクト固有の通信フォーマットがある場合は `RULES.md` で定義する
- スキーマ定義・イベントフォーマット・モジュール間インターフェース仕様は **変更禁止領域**（→ セクション10）

### 2-3. UIとロジックの分離 `[中規模以上]`

- UIコードとビジネスロジックは明確に分離する
- ロジック層はUIに依存しない形で実装する
- UIフレームワークの変更がロジックに影響しない構造を維持する

### 2-4. シークレット管理 `[全規模]`

- パスワード・APIキー・トークン等の機密情報は **平文で保存しない**
- シークレットストアを使用する（具体的な実装は `RULES.md` で指定）
- コード・設定ファイル・ログへの機密情報の埋め込みを禁止する

### 2-5. 外部モジュールのセキュリティクリアランス `[大規模推奨]`

- 外部（サードパーティ）モジュール・ライブラリにはセキュリティクリアランスを適用する
- クリアランスレベルの定義と承認方式は `RULES.md` で規定する
- クリアランスを持たない外部モジュールへの権限付与を禁止する
- 最低限、**信頼済み / 未信頼** の2段階を設けること

## 3. 新規コード作成ルール `[全規模]`

新しいファイル・モジュールを作成する場合、

- ファイル先頭に **検索用コメント（コードインデックス）** を記載する
- 検索用コメントはコードより先に記述する
- 記述形式はセクション5に従う

---

## 4. コード更新ルール `[全規模]`

コードを更新する場合は以下を順に実施する

1. ファイル先頭の検索用コメントを確認する
2. 変更対象の機能インデックスを確認する
3. 変更が他機能・他モジュールに影響しないか確認する
4. 更新後に **変更箇所周辺とヘッダーを** 再確認する
5. 必要に応じて 検索用コメントとインデックスを更新する

### 4-1. AI変更後レビュー規則

変更完了後に以下を必ず確認する

- 変更対象関数だけでなく **同一ファイル全体** を見直す
- 影響を受ける import・型・呼び出し元・テストを確認する
- コメント・関数ID・ログ・型ヒントが変更内容と一致しているか確認する
- AIが追加した未使用コード・暫定コード・デバッグコードを除去する

## 5. 検索用コメント（コードインデックス）仕様 `[全規模]`

### 必須フィールド

```
==================================================
Module / Plugin Name:

Role:
このファイルの役割

Dependencies:
依存する内部モジュール名

Libraries:
使用する外部ライブラリ（例: requests==2.31.0）

Permissions:
必要な権限（プロジェクト定義に従う）

Function Index:
[F01] 機能名 - 1行で機能を説明する
[F02] 機能名 - 1行で機能を説明する

Search Tags:
検索用タグ
==================================================
```

### オプションフィールド（必要に応じて追記）

```
Input:
入力データ

Output:
出力データ

Related Files:
関連ファイル

Security Clearance:
外部モジュールの場合のクリアランスレベル

Change Rules:
変更時の注意事項

Module Stability:
stable / experimental（実験的モジュールはAIが慎重に扱う）

Owner:
Human / AI / Shared（最終変更者の区別）
```

### フォーマット厳守

- `=` の区切り線は10文字以上の `=` のみで構成する
- フィールド名の後には必ず `:` を付ける（スペースなし）
- フィールド値は次のフィールド名が現れるまでの行が対象となる
- `generate_index.bat` のパーサーはこの形式を前提として動作するため、フォーマットの乱れはインデックス生成の失敗につながる

## 6. インデックス自動生成ルール `[中規模以上]`

### 原則

- `index.yaml` はヘッダーコメントから **スクリプトで自動生成** する
- `index.yaml` の手動編集を禁止する（ヘッダーコメントが唯一の真実）
- エージェントはコード探索前に必ず `index.yaml` を参照する

### ファイル構成

プロジェクトルートに以下の2ファイルを配置する

```
generate_index.bat   ← エントリポイント（ダブルクリックでも実行可能）
generate_index.ps1   ← 実処理（bat から呼び出す）
```

### エージェントの実行手順

プロジェクト起動時に毎回実行する

```
1. generate_index.bat を実行する
   → PowerShell が見つからない場合：エラーを表示して終了
   → 実行できない環境の場合：既存の index.yaml をそのまま使う
2. 生成された index.yaml をコンテキストに読み込む
```

### スクリプトファイル

スクリプトの全文は別ファイルを参照すること（プロジェクトルートに配置）。

- `generate_index.bat` ← エントリポイント（PowerShell確認・呼び出し）
- `generate_index.ps1` ← 実処理（ヘッダーパース・index.yaml生成）

### index.yaml の構造

```yaml
generated: '2025-01-01T00:00:00'
root: '/path/to/project'
count: 3
modules:
  - file: plugins/chat_client.py
    name: 'ChatClient'
    role: 'IRC接続とメッセージ受信'
    tags: [irc, websocket, chat]
    dependencies: [stream_monitor]
```

※ AIのトークン消費を最小化するため、関数一覧(functions)、ライブラリ(libraries)、権限(permissions)はインデックスから除外している。詳細が必要な場合は `_index/XX.py.yaml` を参照すること

### _index/ フォルダ（関数詳細インデックス） `[中規模以上]`

`index.yaml` でファイルを特定したあと、さらに **行番号を絞り込む** ために使う。  
ファイル全体を読まずに該当関数の行範囲だけをロードできるため、大きなファイルでのトークン消費を大幅に削減できる。

**配置場所:** プロジェクトルートの `_index/` フォルダ（自動生成・手動編集禁止）

**ファイル命名規則:** ソースファイルの相対パスのスラッシュ・バックスラッシュを `__` に変換してフラット化し、末尾に `.yaml` を付加する

```
例: plugins/chat_client.py  →  _index/plugins__chat_client.py.yaml
    core.py                 →  _index/core.py.yaml
```

**XX.py.yaml の構造:**

```yaml
file: 'plugins/chat_client.py'
generated: '2025-01-01T00:00:00'
functions:
  - id: F01
    name: connect
    desc: WebSocket接続を確立してセッションを返す
    line_start: 45
    line_end: 78
  - id: F02
    name: disconnect
    desc: セッションを切断してリソースを解放する
    line_start: 80
    line_end: 95
```

**生成スクリプト（generate_index.ps1）の動作:**

```
1. 全対象ファイルをスキャンして index.yaml を生成（従来通り）
2. index.yaml に記載された各ファイルの [Fxx]〜[FEND] ブロックを解析して _index/XX.py.yaml を生成・更新
3. _index/ に存在するが index.yaml にないファイルの .yaml を自動削除
```

## 7. 機能インデックスルール `[全規模]`

各機能にはIDを付与する。形式は `[Fxx] 関数名 - 1行の機能説明` とする

```
[F01] load_config  - 設定ファイルを読み込んでdictで返す
[F02] get_auth     - 環境変数からAPIキーを取得して検証する
[F03] connect      - WebSocket接続を確立してセッションを返す
[F04] recv_message - 受信ループを起動してコールバックに渡す
```

関数の直前に同じIDを記載し、関数の終端に `# [FEND]` を記載する

```python
# [F03] connect - WebSocket接続を確立してセッションを返す
def connect():
    ...
# [FEND]
```

`[FEND]` を付与することで `generate_index.ps1` が行番号を正確に記録し、  
AIエージェントが `_index/XX.py.yaml` を参照して **該当行範囲のみ** を読み込める。

### 7-1. ネスト・デコレータの扱い

- クラス内のメソッドも同様に `[Fxx]` / `[FEND]` を付ける
- デコレータがある場合は **デコレータ行の直前** に `# [Fxx]` を置く
- `__init__` もFIDの対象とする（例: `[F01] __init__ - インスタンスを初期化する`）
- 関数内にネストした内部関数には **外側の関数のみ** FIDを付ける（内部関数は不要）

### 7-2. 運用ルール

- 重要関数には必ず関数IDを付ける
- 関数IDはファイル内で一意にする
- 大きな責務変更時はIDを維持し、役割が完全に変わる場合のみ採番し直す
- レビューや修正依頼では関数名だけでなく関数IDも併記する（例：「[F03] connect を修正」）
- ヘッダーの `Function Index` と `# [Fxx]` マーカーおよび `_index/XX.py.yaml` の内容は常に一致させる

## 8. ログ出力ルール `[全規模]`

すべての主要処理はログを出力する

### ログに含める内容

- 処理開始 / 処理終了
- エラー
- 再接続
- 外部通信
- 重要な状態変更
- セキュリティクリアランスの判定結果

### ログ形式

```
[INFO]  module_name : message
[WARN]  module_name : message
[ERROR] module_name : message
```

### ログファイルの出力規則

ログは以下の 4フォルダ に同時出力する。各フォルダは ルートに最新ファイル、History/ に過去ファイル を置く統一構造とする。

```
logs/
  ├── {app_name}.log               ← AI最優先読み取り（上書き）
  └── History/
        └── {app_name}_{timestamp}.log

runtime/
  ├── {app_name}.log               ← AI詳細読み取り・必要時（上書き）
  └── History/
        └── {app_name}_{timestamp}.log

All-Logs/
  ├── {app_name}.log               ← 人間・最新全行動（上書き）
  └── History/
        └── {app_name}_{timestamp}.log

AI-Logs/
  ├── {app_name}.log               ← 作業工程・思考・引き継ぎ（AI用思考ログ）
  └── History/
        └── {app_name}_{timestamp}.log
```

| フォルダ | 読む主体 | 内容 | レベル |
|---|---|---|---|
| `logs/` | AIエージェント（最優先） | 最小限。通常時はOK/FAIL+件数+run_id、エラー時のみ種別・箇所・直前操作5件 | WARNING以上 |
| `runtime/` | AIエージェント（必要時） | 最新1回の詳細ログ・スタックトレース含む | INFO以上 |
| `AI-Logs/` | AIエージェント（文脈確認） |作業工程・思考・引き継ぎ（AI用思考ログ）（5世代管理） | -|
| `All-Logs/` | 人間（事後確認専用） | 全行動。全ステップ・入出力・判断・ツール呼び出し結果 | DEBUG以上 |

#### AIの読み取り順序

```
1. logs/{app_name}.log          必須・最優先
2. AI-Logs/{app_name}.log              前回の作業内容や未完了タスクの文脈を把握。
3. runtime/{app_name}.log       必要時のみ
4. logs/History/ または runtime/History/   履歴確認時のみ
5. All-Logs/                    詳細解析時のみ（原則読まない）
```

#### run_id の必須化

すべてのログに `run_id` を付与する。形式: `YYYYMMDD_HHMMSS_NNN`

```
[WARN] app : process_complete : total=42 : run_id=20260318_153012_001
```

- @Why: 複数回実行時にログが混在せず、エージェントが該当セッションを一意に追跡できる

### 注意

- ログに機密情報（トークン・パスワード等）を含めない
- ログの詳細形式はプロジェクトの `RULES.md` で定義する

## 14. エージェントの自律実行に対する安全基準（Guardrails） `[全規模]`

### 14-1. エラー発生時の回復手順

- エラーが発生した場合、推測による修正（当てずっぽうなコード変更）を禁止する
- 必ずログやスタックトレースを出力・解析し、原因を特定してから修正案を提示すること

### 14-2. アトミックな変更とチェックポイント

- 複数の機能を一度に修正・実装しないこと
- `Function Index` に基づき、1つのタスク（例: [F01]）が完了し動作確認が取れた段階で区切りを設ける

### 14-3. 人間の介入（Human-in-the-Loop）条件

以下の操作はエージェントの単独実行を禁止し、必ず事前に人間の承認を得ること

- ファイルの削除、または大規模な書き換え
- データベーススキーマの破壊的変更
- 認証情報や環境変数の書き換え
- 外部APIへの破壊的リクエスト
- 外部ライブラリの新規追加（→ セクション18で詳細規定）

## 15. コーディング規約 `[全規模]`

### 15-1. 型定義の義務化

- 動的型付け言語（Python等）であっても、原則として型ヒント（Type Hints）を明記すること
- モジュール間のJSON通信におけるスキーマ齟齬や、データ構造のハルシネーションを未然に防ぐため

### 15-2. 冪等性（Idempotency）の確保

- 初期化処理や状態変更を伴うスクリプトは、何度実行しても同じ結果になるよう実装すること
- エージェントの再試行ループによってシステムやデータが破壊されるのを防ぐため

### 15-4. AI向けメタコメントの導入

複雑なロジックや特殊な実装を行った場合、コード内にエージェント向けの理由コメントを残すこと

**形式**

```python
# @AI-Note: <なぜこの実装にしたか・推論・注意点>
```

**例**

```python
# @AI-Note: リトライをここで行う理由は接続断が一時的なケースが多いため。
#            最大3回・指数バックオフ。上限を超えたら呼び出し元にエラーを返す。
def connect_with_retry():
```

セッションリセット時や他のエージェントが引き継いだ際のコンテキスト喪失を防ぐため、
`@AI-Note:` プレフィックスを統一して使用すること（検索性を維持するため）

### 15-5. 処理説明コメントの導入

コードを読まなくても意図が追えるよう、以下のプレフィックスを使ってコメントを記載する。

バイブコーディングの普及により「動くけど読めない」コードが増えている。
`@What:` と `@Why:` はコードの意図を人間にもAIにも伝えるためのインフラとして機能する。

| プレフィックス | 目的 | 書く場所 |
|---|---|---|
| `@What:` | 処理の説明・言語構文の解説・他言語との対応 | 複雑な処理・言語固有の書き方の直前 |
| `@Why:` | 設計理由・判断の根拠 | 関数・重要なロジックの直前 |
| `@AI-Note:` | エージェント向けの注意・制約 | 修正時に知っておくべき箇所 |

**例**

```python
# @What: リストから条件に合う要素だけを取り出す
#        Python のリスト内包表記。他言語の filter() / where() に相当する
active_users = [u for u in users if u.is_active]

# @Why:  接続失敗を呼び出し元に委ねるため例外をそのまま投げる
#        ここで握りつぶすと上位のリトライ処理が機能しなくなる
# @AI-Note: タイムアウト値は設定ファイルから取得すること。ハードコーディング禁止
def connect():
```

**記載の基準**

- すべての行にコメントを書く必要はない
- 言語固有の構文・初見でわかりにくいロジック・設計上の判断が入る箇所を優先する
- `@What:` と `@Why:` は必要に応じて片方だけでもよい

---

### 15-3. テストの義務化と自己検証 `[中規模以上]`

- 原則として実装前にテストコード（または検証用スクリプト）を作成し、
  **意図的に失敗することを確認してから実装に移る**（TDD推奨）
- コードを変更した後は該当機能のテストを必ず実行する
- テスト成功を確認してからタスクを完了とみなす
- テスト未実行状態で作業完了と判断してはならない
- 具体的なテストフレームワーク・実行手順は `RULES.md` で定義する

## 16. バージョン管理（Git）ルール `[中規模以上]`

> **適用条件**: チーム開発またはリリース済みプロジェクトでは必須。  
> 個人開発・プロトタイプ段階では推奨とし、適用レベルは `RULES.md` で明示すること。

### 16-1. ブランチの分離と保護

- メインブランチ（`main` または `master`）への直接コミットを禁止する
- 作業着手前に必ず作業ブランチを作成すること

```
feature/機能名      ← 新機能
fix/修正内容        ← バグ修正
refactor/対象名     ← リファクタリング
```

### 16-2. アトミックなコミット

- コミットは「安全にロールバックできる最小の機能単位」で行うこと
- 複数の異なる変更を1つのコミットに混ぜない

### 16-3. 自己説明型のコミットメッセージ

コミットメッセージには以下を含めること

- 変更対象の `Function Index` またはモジュール名
- 何を変更したか
- **なぜその変更を行ったか（理由・推論）**

**形式**

```
<type>([対象]): <変更内容>。<理由>。

例:
feat([F02]): 認証情報取得ロジックにリトライ処理を追加。
             一時的なネットワーク切断時のクラッシュを防ぐため。
```

---

### 16-4. コンフリクト解決手順

マージコンフリクトが発生した場合：

- AIによる推測での自動解消を禁止する
- コンフリクトマーカー（`<<<<<<<` 等）周辺の状況と両ブランチの変更意図を整理する
- 解決方針を人間に報告し、指示を仰いでから解消する

```
# コンフリクトマーカーの読み方
<<<<<<< HEAD（自分のブランチの変更）
コードA
=======
コードB
>>>>>>> branch-name（マージしようとしたブランチの変更）
```

- コンフリクトマーカーだけ除去してどちらの変更も反映しないことを禁止する
- 推測で片方を選んで消すことを禁止する

## 19. コーディングスタイルの統一 `[全規模]`

言語固有の詳細規約は `RULES.md` で定義する。以下は言語をまたいで適用する原則。

### 19-1. 関数・メソッドの設計

- 1関数1責務を維持する
- 関数は短く保つ（目安：50行以内。超える場合は分割を検討）
- 引数は少なく保つ（目安：4個以内）

### 19-2. 可読性

- マジックナンバーは定数・設定値として定義する
- ネストは3段以内を目安とする。深くなる場合は早期リターンや関数分割を使う
- 略語・省略形の使用を避け、意図が伝わる名前をつける

### 19-3. 不要コードの禁止

- コメントアウトしたコードを残さない（Gitで管理するため不要）
- 未使用の変数・関数・インポートを残さない
- デバッグ用のprint・console.log等を本番コードに残さない

## 20. 品質方針 `[全規模]`

### 20-1. エラー処理

- エラーを握りつぶすことを禁止する（空のcatch・except節を禁止）
- エラーは必ずログに記録し、適切に上位へ伝播させること
- エラーメッセージには原因と発生箇所を含めること

### 20-2. 副作用の明示

- 外部状態（ファイル・DB・ネットワーク・グローバル変数）を変更する関数は、
  関数名または `@AI-Note:` でその副作用を明示する

### 20-3. ハードコーディングの禁止

- URLやファイルパス・設定値をコード内にハードコーディングしない
- 設定ファイル・環境変数・定数定義に切り出すこと

### 20-4. コードレビューの基準

エージェントが実装後に自己チェックする項目：

- ヘッダーコメントは最新の実装と一致しているか
- 型ヒントは付与されているか
- エラー処理は漏れていないか
- ハードコーディングはないか
- テストは実行したか

### 20-5. 外部データの非信頼原則（ゼロトラスト） `[全規模]`

- ネットワーク・ユーザー入力・外部ファイル（APIレスポンス・チャットコメント等）から
  取得したデータはすべて「非信頼データ」として扱う
- 外部データをログ出力・DB保存・LLMプロンプトへ渡す際は必ずサニタイズまたはエスケープ処理を実装する
- 外部入力をそのまま `eval()` 系関数で評価すること・システムコマンドとして実行することを禁止する

## 21. ファイル命名規則 `[全規模]`

言語固有の慣習がある場合は `RULES.md` で上書きする。以下は共通原則。

### 21-1. 基本原則

- ファイル名は **役割が一目でわかる名前** にする
- 略語・連番（`util2.py`、`module_new.js`）を禁止する
- 英数字・アンダースコア・ハイフンのみ使用する（スペース禁止）

### 21-2. 命名パターン

| 種別 | パターン | 例 |
|---|---|---|
| モジュール・クラス | `snake_case` または `PascalCase`（言語慣習に従う） | `chat_client.py` / `ChatClient.cs` |
| 設定ファイル | `snake_case` | `app_config.toml` |
| テストファイル | `test_対象名` または `対象名_test` | `test_chat_client.py` |
| スクリプト | `動詞_対象` | `generate_index.ps1` |
| ドキュメント | `SCREAMING_SNAKE_CASE`（重要文書）または `snake_case` | `GLOBAL_RULES.md` / `api_reference.md` |

### 21-3. ディレクトリ構成の原則

- ディレクトリ名は役割を表す名詞にする
- 深すぎる階層を避ける（目安：4階層以内）
- プロジェクト固有のディレクトリ構成は `RULES.md` で定義する

### 21-4. ファイルサイズと認知負荷の管理 `[中規模以上]`

- 1ファイルの行数が **500行（目安）を超える**、または今回の修正で超えると予測される場合、
  そのまま追記することを原則禁止とする
- 新規実装に入る前に、現在のファイルをどのように複数モジュールに分割すべきか
  計画書を作成して人間に提案する
- AIのコンテキストウィンドウ枯渇と全体構造のスパゲッティ化を未然に防ぐための措置である

## 22. AI探索・コンテキスト節約ルール `[全規模]`

### 22-1. 探索順序

エージェントはコードを探索する際、以下の順序を厳守する

```
1. index.yaml を確認する（対象ファイルを特定）
2. _index/XX.py.yaml を確認する（Function IDと行番号を特定）
3. 対象ファイルの該当行範囲のみを読む（行番号指定で最小取得）
4. ヘッダーコメントの確認が必要な場合は先頭部分のみ読む
5. ファイル全文を読むのは最後の手段とする
```

**行範囲指定の例（PowerShell）:**

```powershell
# XX.py.yaml で line_start=45, line_end=78 と分かった場合
Get-Content chat_client.py | Select-Object -Skip 44 -First 34
```

### 22-2. 探索の禁止事項

- 最初から全ファイル走査をしてはならない
- 最初から全文読みに入ってはならない
- まず「どのファイルのどの機能を見るべきか」を特定してから読む
- 修正対象に直接関係しないファイルは読まない
- 依存関係が不明な場合のみ `Dependencies` をたどって追加調査する

### 22-3. コンテキスト消費の最小化

- 全文読み込みを標準動作にしてはならない
- 修正前に「必要最小限の範囲」を特定する
- 一度読んだ内容は要約して内部メモ化し、同じ範囲の再読を減らす
- 変更に関係しない定数群・補助コードは読み飛ばす
- 出力時も必要な差分・要点・影響範囲を優先し、冗長な全文再掲を避ける

### 22-4. プロンプトキャッシング最適化 `[全規模]`

> エージェントを呼び出すシステムを構築する側が配慮する設計ルール。

- `PROJECT_CONFIG.md`・`index.yaml`・共通ルールなど「変化しない・変化頻度が低い情報」は
  プロンプトの **先頭** に固定して配置する
- 毎回のタスク指示や動的に変わる情報はプロンプトの **末尾** に配置する
- これによりKVキャッシュのヒット率を最大化し、処理速度向上とコスト削減を図る

### 22-5. AST・スケルトン探索の推奨 `[大規模推奨]`

- 未知の巨大なモジュール群を探索する際は、最初からフルテキストを読み込まない
- 利用できる場合はAST（抽象構文木）パーサーや Tree-sitter 等を活用し、
  関数シグネチャとヘッダーコメントのみを抽出した「スケルトン」で全体像を把握する
- 実装の詳細が必要と判断した特定の `[Fxx]` ブロックのみフルテキストを取得する
- ツールが利用できない場合は index.yaml と Function Index で代替する

## 23. 設計記録ルール `[中規模以上]`

重要な設計判断は永続ドキュメントに記録し、セッションが変わっても参照可能にする。

### 23-1. 記録の原則

- 重要な判断を会話だけで終わらせない
- 「なぜその構成にしたか」を必ず残す
- 破壊的変更・方針転換・ライブラリ採用理由は記録する
- 次のAI・次回セッションが読めば再開できる状態を保つ
- 実装前の方針と実装後の差分も必要に応じて残す

### 23-2. 推奨ドキュメント

プロジェクトの規模・性質に応じて以下から必要なものを作成する。具体的なファイル構成は `RULES.md` で定義する。

| ファイル | 内容 |
|---|---|
| `CONTEXT.md` | 現在の設計状態・制約・未解決問題（作業開始前に必ず確認） |
| `DESIGN.md` | 全体設計・アーキテクチャ方針 |
| `DECISIONS.md` | 設計判断の履歴・採用理由 |
| `KNOWN_ISSUES.md` | 既知の問題と暫定対応 |
| `ROADMAP.md` | 今後の実装計画 |
| `CHANGELOG.md` | 機能追加・変更の履歴（ユーザー・AI向け） |

### 23-3. ユーザー向け変更履歴（CHANGELOG）の維持 `[中規模以上]`

- `[Fxx]` の追加や機能の仕様変更を伴うタスクが完了した場合、
  `{app_name}.log` だけでなく `CHANGELOG.md` も更新する
- コミットログのような技術的詳細ではなく、
  「人間および別のAIが読んでどのような機能が増えたか・変わったか」が分かる粒度で記載する

## 24. AI作業ログルール `[中規模以上]`

エージェントが行った作業は `{app_name}.log` にチェックポイントごとに記録する。

### 24-1. フォルダ構造と世代管理

他のログフォルダと同様、最新ファイルと履歴フォルダの二段構成で管理する。

AI-Logs/
  ├── {app_name}.log                ← 最新の作業ログ（追記型）
  └── History/
        └── ai_log_{timestamp}.log

-最新ログの保持: {app_name}.log には直近の作業内容を追記していく。
-履歴への移動（ローテーション）: 以下のいずれかの条件を満たした場合、内容を History/ に退避させ、{app_name}.log を新規作成（クリア）する。

   -およそ 5回分（5セッション分） の作業ログが蓄積された場合。
   -ファイルサイズが肥大化し、AIのコンテキストウィンドウを圧迫し始めた場合。
   -プロジェクトの大きなマイルストーンや機能実装が完了した場合。

### 24-2. 記録タイミング

作業完了後にまとめて書くのではなく、以下のタイミングで随時記録する

- 作業開始時（何をするか・対象ファイル）
- Function Index 単位の作業完了時（何が終わったか）
- エラー発生時（何が起きたか・暫定対応）
- セッション終了時・クオータ切れ直前（次に何をすべきか）

リアルタイム記録により、強制終了・クラッシュ・クオータ切れのどのタイミングで止まっても状態が復元できる。

### 24-3. 記録内容

各記録の冒頭には、セッションを一意に特定するための run_id を付与することを推奨する。

```
## [日時] 作業ログ(run_id: YYYYMMDD_HHMMSS_NNN)

### 作業開始
- 担当     : AI / Human
- 対象ファイル：
- 実施内容：

### 完了チェックポイント
- [F01] 完了 / 未完了
- [F02] 完了 / 未完了

### エラー・問題
- 発生内容：
- 試したこと：
- 暫定対応：

### 次回引き継ぎ
- 次にすべきこと：
- 未完了事項：
- 注意点：
```

### 24-4. ルール

- 失敗した試行・却下した案も必要に応じて記録する
- 別のAIや人間が読んで状況把握できる粒度で書く
- `{app_name}.log` はGitで管理し、履歴を保持する

### 24-5. コンテキスト劣化時の撤退ルール

以下のいずれかに該当する場合、推測での修正をただちに停止する

- 同じエラーが連続して3回以上解決しない
- タスクが長引きコンテキストの混乱（ハルシネーション）を自覚した

停止後の手順：

```
1. その時点での状況・試したこと・行き詰まっている原因を {app_name}.log に書き出す
2. 人間（または別エージェント）にセッションの再起動と介入を要求する
3. 推測での修正・応急処置的なコード変更を行わない
```

- 「もう少しで解決できる」という判断で推測修正を続けることを禁止する
- 撤退は失敗ではなく、コードとコンテキストを守る正しい判断である

### 24-6. 長期セッションにおけるコンテキストの要約と破棄 `[中規模以上]`

- セッションの会話履歴が長くなりコンテキストウィンドウの枯渇が近づいた場合、
  過去のやり取りをそのまま保持し続けてはならない
- 「直近のタスクに関連する文脈」のみを要約して保持し、
  古い実行ログや不要になった探索の記憶は意図的に破棄（スライディングウィンドウ）する
- 重要な決定事項は `{app_name}.log` や `DECISIONS.md` に書き出し、コンテキストを身軽に保つ

## 25. 環境非依存ルール `[全規模]`

コードが特定の環境に依存してエラーになることを防ぐ。
言語バージョン依存は各言語ルール（例: `PYTHON_RULES.md`）で定義する。

### 25-1. 文字コード

- コード・ファイル・通信はすべて **UTF-8 に統一する**
- エンコーディングを明示せずにファイルを開くことを禁止する
- OSのデフォルトエンコーディングに依存しない

```python
# ✅ 推奨
with open(path, encoding="utf-8") as f: ...

# ❌ 禁止（OSのデフォルトに依存する）
with open(path) as f: ...
```

### 25-2. タイムゾーン

- 日時データは **UTC で保存・通信する**
- 表示時のみ OS のタイムゾーンに変換する
- タイムゾーンを明示しない naive な日時オブジェクトの使用を禁止する

```
保存・通信 → UTC（環境非依存）
表示       → OSのタイムゾーンに変換
```

### 25-3. 環境変数

- コードが環境変数に直接依存することを禁止する
- 設定ファイルを経由して取得し、環境変数はフォールバックとして使う
- 環境変数が存在しない場合のデフォルト値を必ず定義する

```
優先順位：設定ファイル → 環境変数 → デフォルト値
```

- APIキー等の機密情報はシークレットストアを使う（→ セクション2-4）

### 25-4. ハードウェア

- CPUコア数・メモリ量・GPU有無をハードコーディングしない
- ハードウェアスペックは動的取得または設定値経由で取得する
- GPU等の特殊ハードウェアを使う場合は CPUフォールバックを用意する

```
CPUコア数  → os.cpu_count() 等で動的取得
GPU有無    → 条件分岐でCPUフォールバックを実装
メモリ量   → 設定値で調整できる設計にする
```

- ハードウェア依存の処理を使う場合は `@AI-Note:` で依存内容と代替手段を明記する

### 25-5. ファイルパス

- ファイルパスの区切り文字をハードコーディングしない
- 言語・フレームワークが提供するパス操作APIを使う
- 絶対パスをハードコーディングしない

```python
# ✅ 推奨（Python）
from pathlib import Path
config_path = Path(__file__).parent / "config" / "app.toml"

# ❌ 禁止
config_path = "C:\\Users\\user\\project\\config\\app.toml"
config_path = "./config/app.toml"  # 実行場所依存
```

- プロジェクトルートからの相対パスが必要な場合は `PROJECT_CONFIG.md` に記載する

## 26. Agent Skills ルール `[全規模]`

### 26-1. 基本方針

- スキルはエージェントが「必要な時だけ読む」指示書単位で管理する
- スキルフォルダは `PROJECT_CONFIG.md` で指定する（デフォルト: `skills/`）
- スキルの一覧は `skill_index.yaml` で管理する（自動生成・手動編集禁止）
- エージェントは起動時に必ず SkillManager を最初に読む

### 26-2. SkillManager

SkillManager はすべてのスキルを管理するメタスキルである。

```
skills/
  skill_index.yaml          ← スキル一覧（自動生成）
  generate_skills.bat       ← エントリポイント
  generate_skills.ps1       ← 実処理（SKILL.mdを走査してjson生成）
  skill_manager/
    SKILL.md                ← 常に最初に読む
    skill_manager.md        ← 検索・選択手順
```

**エージェントの起動手順**

```
1. skills/skill_manager/SKILL.md を読む
2. skill_index.yaml を読む
3. 作業内容に関連するスキルを tags・description で特定する
4. 該当スキルのみ読み込む
5. 関係しないスキルは読まない
6. スキルで解決しない場合のみ GLOBAL_RULES.md を参照する
```

### 26-3. SKILL.md フォーマット

各スキルフォルダに `SKILL.md` を配置する。先頭にYAMLフロントマターで定義する。

```yaml
---
name: HeaderComment
description: ヘッダーコメントを書く・確認する時に使う
priority: 1
tags: [header, コメント, 作成]
---

# スキルの詳細内容をここに書く
```

| フィールド | 内容 |
|---|---|
| `name` | スキル名（一意） |
| `description` | いつ使うか（エージェントが判断に使う） |
| `priority` | 読み込み優先度（SkillManagerは0・他は1以上） |
| `tags` | 検索用タグ（配列） |

### 26-4. skill_index.yaml の構造

```json
{
  "generated": "2025-01-01T00:00:00",
  "skills_root": "I:/.agents/MainSkills/skills",
  "count": 3,
  "skills": [
    {
      "name": "SkillManager",
      "path": "skill_manager/SKILL.md",
      "description": "スキルの検索・管理を行う。常に最初に読み込む。",

    {
      "name": "HeaderComment",

      "priority": 1,
      "tags": ["header", "コメント", "作成"]
    }

```

### 26-5. スキル自動生成ルール

- スキルを追加・変更した後は必ず `generate_skills.bat` を実行する
- `skill_index.yaml` は手動編集禁止（ヘッダーの `SKILL.md` が唯一の真実）
- `skill_index.yaml` は変更禁止領域に含む（→ セクション11）

### generate_skills.bat

```bat

chcp 65001 > nul
echo [INFO] generate_skills : starting...

where powershell >nul 2>&1
if %errorlevel% neq 0 (
    echo [ERROR] generate_skills : PowerShell not found.
    pause
    exit /b 1
)

powershell -ExecutionPolicy Bypass -File "%~dp0generate_skills.ps1" %*
if %errorlevel% neq 0 (
    echo [ERROR] generate_skills : script failed.
    pause
    exit /b 1
)

```

### generate_skills.ps1

```powershell
# generate_skills.ps1
# SKILL.md を走査して skill_index.yaml を生成する
# 使用方法: .\generate_skills.bat [-Root <path>] [-Out <path>]

param(
    [string]$Root = ".",
    [string]$Out  = "skill_index.yaml"
)

function Parse-SkillHeader {
    param([string]$FilePath)

    $content = Get-Content $FilePath -Raw -Encoding UTF8 -ErrorAction SilentlyContinue
    if (-not $content) { return $null }

    # YAMLフロントマターを抽出
    if ($content -notmatch "(?s)^---\s*\n(.+?)\n---") { return $null }
    $yaml = $Matches[1]

    $fields = @{}
    foreach ($line in $yaml -split "\n") {
        if ($line -match "^(\w+):\s*(.+)$") {
            $key   = $Matches[1].Trim()
            $value = $Matches[2].Trim()
            $fields[$key] = $value
        }
    }

    return $fields
}

function Parse-Tags {
    param([string]$Raw)
    # [tag1, tag2] 形式をパース
    $cleaned = $Raw -replace "[\[\]]", ""
    return ($cleaned -split "[,\s]+") | Where-Object { $_ }
}

$skills = @()

Get-ChildItem -Path $Root -Recurse -Filter "SKILL.md" -ErrorAction SilentlyContinue |
Where-Object { $_.FullName -notmatch "\\(\.git|__pycache__|node_modules)\\" } |
ForEach-Object {
    $fields = Parse-SkillHeader $_.FullName
    if (-not $fields) { return }

    $relPath = $_.FullName.Replace((Resolve-Path $Root).Path, "").TrimStart("\", "/")

    $skills += [ordered]@{
        name        = $fields["name"]
        path        = $relPath -replace "\\", "/"
        description = $fields["description"]
        priority    = [int]($fields["priority"] ?? 1)
        tags        = @(Parse-Tags ($fields["tags"] ?? ""))
    }
}

# priority 昇順でソート
$skills = $skills | Sort-Object { $_["priority"] }

$output = [ordered]@{
    generated   = (Get-Date -Format "yyyy-MM-ddTHH:mm:ss")
    skills_root = (Resolve-Path $Root).Path -replace "\\", "/"
    count       = $skills.Count
    skills      = $skills
}

$output | ConvertTo-Json -Depth 10 | Set-Content $Out -Encoding UTF8
Write-Host "[INFO] generate_skills : generated $($skills.Count) skills -> $Out"
```

### 26-6. 複数エージェントの共有

- スキルフォルダはすべてのエージェントで共有する
- 標準パス: `I:\.agents\MainSkills\skills\`
- 各エージェントのスキルフォルダをこのパスに向ける
  - パス変更できる場合 → エージェントの設定ファイルで指定する
  - できない場合 → `mklink /D "[エージェントのスキルフォルダ]" "I:\.agents\MainSkills\skills\"` でシンボリックリンクを作成する

### 26-7. スキルと分割ルールファイルの英語化推奨 `[中規模以上]`

- エージェントが読む分割ルールファイル（`rules/` 以下）および `skills/` 以下の `SKILL.md` は
  英語で記述することを推奨する
- トークン消費を最小化し、英語圏ユーザーへの配布時の障壁を除くため
- ルールファイルが英語であっても、エージェントへの出力指示は日本語に設定する
  （例: `Output strictly in Japanese.` をCORE.mdに明記）
- 正本ファイルの管理言語：
  - 日本語正本（`GLOBAL_RULES.md` 等）← 編集・管理の基準
  - 英語正本（`GLOBAL_RULES_EN.md` 等）← 英語圏ユーザーへの配布用
  - 英語分割版（`rules/` 以下）← エージェントが読む圧縮版
